AIDEEN TOKEN CIRCUIT TRACE
==========================

Purpose
-------
This document is the authoritative living trace of the token lifecycle through AIDEEN.
It must be updated whenever the model path, memory path, training path, GPU layout, or debugging understanding changes.

Rule for completeness
---------------------
For every variable/function/buffer/env flag mentioned here, all transitive definitions, writes, reads, layout assumptions, and CPU/GPU crossings must be searched and understood before declaring the path stable.

0. Sequence construction
------------------------
File/function: aideen-training-lab/src/bin/assoc_recall.rs::make_ar_sequence
Role:
- Creates associative recall sequences.
- Format for pairs=1: KEY_i VAL_i FILLER*gap KEY_i QUERY VAL_i.
- Format for pairs>1: (KEY_j VAL_j)*N FILLER*gap KEY_q QUERY VAL_q.
- The final VAL token is the answer target.

Important tokens:
- KEY tokens: TOK_KEY_BASE + key.
- VAL tokens: TOK_VAL_BASE + val.
- FILLER tokens: TOK_FILLER_BASE + random filler.
- QUERY token: TOK_QUERY.

Invariant:
- The desired durable binding is previous KEY token -> current VAL token for the initial pair section.
- Filler transitions should not overwrite or diffusely contaminate finite associative banks.

Open trace tasks:
- Document exact token constants and ranges after inspecting definitions.
- Trace whether any marker semantics are available to GPU shaders or only to benchmark code.

5. Forward associative read
---------------------------
File/shader: aideen-block/src/shaders/deq_slot_attn_unified_clean.wgsl
Buffers:
- AssocBuf: durable bank memory, per slot and bank.
- AssocReadBuf: selected/read context diagnostics or read state.
- AllWeights: W_q_assoc, alpha_assoc, slot anchors, etc.

Conceptual equation:
- Query is built from previous associative source source_{t-1}.
- Banks contain key/value/usage.
- Read selects bank(s) by similarity between query and bank key.
- Read context contributes to token representation/logit path depending active flags.

Important diagnostics:
- qmax, qctx, usage, spread, bank_cos, chosen_read_prob, target logit/prob.

Observed failure:
- Read often chooses occupied banks with nontrivial probability, but bank value/logit remains near random because bank contents are diffuse.

Open trace tasks:
- Trace exact read contribution into H/Scratch and LM output.
- Determine whether LM head receives Assoc value directly, via h*, via late fuse, or diagnostic-only route for each flag.

6. Forward associative write
----------------------------
File/shader: aideen-block/src/shaders/deq_slot_attn_unified_clean.wgsl
Buffers:
- PrevHStarBuf: previous token literal source.
- AssocBuf: bank key/value/usage.
- AssocHist: pre-write bank snapshot plus metadata for backward.
- AllWeights: W_k_assoc, W_v_assoc, W_event_assoc, b_event_assoc, slot anchors.

Current conceptual equation:
- event_gate = sigmoid(W_event * feature(source_{t-1}, source_t) + b_event) when ENABLE_ASSOC_EVENT_GATE.
- bind_gate = event_gate * has_prev_hstar.
- slot allocator gives p_slot_i.
- bank allocator gives p_bank.
- allow_write = event_gate * p_slot_i * p_bank.
- effective_bind_gate = bind_gate * allow_write * transition_gate.
- effective_write_mass = effective_bind_gate^2 unless linear write flag is active.
- bank_key <- (1 - mass) bank_key + mass * key(source_{t-1}).
- bank_value <- (1 - mass) bank_value + mass * value(source_t).
- usage updates with write mass.

Known failure:
- Long gaps still produce writes across filler tokens; finite banks become an EMA/mix of many tokens.
- w_n remains proportional to sequence length in diagnostics.

Known structural fix pending validation:
- Forward now receives ENABLE_ASSOC_EVENT_GATE and related constants, matching backward.

Open trace tasks:
- Validate that event_b changes now reduce forward w_allow/w_gate.
- If not, trace event feature, parameter offsets, and AllWeights indexing.

AssocHist segmented coordinate invariant
----------------------------------------
Forward file: aideen-block/src/shaders/deq_slot_attn_unified_clean.wgsl
Backward file: aideen-backbone/src/shaders/fused_fpm_retain_bwd.wgsl
Buffer owner: RustDeqBridge.assoc_hist_buf, allocated as [batch][max_seq_len][h_slots][assoc_hist_slot_stride].
Forward token coordinates: local segment loop variable `t`, full-sequence token `global_t = shape.token_start + t`, full-sequence length `shape.seq_len`.
Backward token coordinates: `t` over `params.seq_len`, absolute index `t_abs = b * params.seq_len + t`.
Required invariant: every AssocHist write for a forward token must use the full-sequence coordinate `(batch_idx * shape.seq_len + global_t)`, because backward reads `(b * params.seq_len + t)`.
Failure mode found: local segment indexing can overwrite early history positions and leave later positions stale when internal segmented forward is active.


Stabilized Assoc multi-binding path update
------------------------------------------
- Assoc address path now treats `W_k_assoc`/`W_q_assoc` as an optional learned residual over the direct normalized token/source address. `AIDEEN_ASSOC_ADDR_LR_MULT` defaults to 0.0, preserving the literal key/query manifold during current explicit-memory training.
- When `ENABLE_ASSOC_SLOT_OWNER=false`, Assoc slots are replicated lanes, not competitors. Each active Assoc slot writes full local binding mass. If slot ownership is enabled later, `p_slot_i` returns to normalized owner probability.
- Bank allocation now uses empty-first placement for novel bindings: if a free bank exists and no existing key match is strong enough, the new binding is committed to the free bank instead of being dropped by occupied-bank protection.
- Assoc retrieval is winner-take-all by default. Soft bank mixtures are experimental because they blend multiple literal values and contaminate exact key->value recall. FPM remains the graded/contextual memory path.
- Validated default AR corridor: pairs_per_seq=8, banks=8, gaps 0/64/128/256 solved with losses around 0.0002-0.0004.

Assoc scalable capacity note - adaptive hash replication
-------------------------------------------------------
Current experimental capacity path:
- `AIDEEN_ASSOC_HASH_LANE=1` makes Assoc lane ownership deterministic rather than fully competitive.
- `AIDEEN_ASSOC_HASH_REPLICA=1` can add a deterministic adjacent lane as a second probe, but only when `ASSOC_BANKS >= 16` in the shader path.
- For `ASSOC_BANKS < 16`, replica is disabled even if the env flag is set, preserving the single-owner small-bank circuit.

Circuit invariant:
- Write and read must share the same owner/probe lane set for a binding.
- Replication is a collision-reduction mechanism, not a replacement for salience or durable slot policy.
- Literal Assoc recall remains separate from FPM contextual consolidation: Assoc stores explicit key->value evidence; FPM may summarize/contextualize but should not be required to recover literal pairs that Assoc failed to preserve.

Validated observations:
- p16/b8 is protected by the adaptive guard and remains in the working small-bank corridor.
- p24/b16 can work with replica, proving that the previous p24 failure was largely capacity/collision/routing related.
- p24/b16 variance shows the next unresolved circuit segment is write salience/replacement under lane pressure.

Capacity edge observation
-------------------------
- h8/b16 with adaptive two-lane Assoc replication solved p32 (`eval_loss=0.0941`, random=4.4067).
- This means the current token path can preserve many literal bindings when lane/bank geometry is sufficient.
- The unresolved problem is now narrower: behavior when requested bindings exceed effective lane/bank capacity, or when salience should decide which bindings persist across longer documents/segments.

LM sampled-softmax probability invariant
----------------------------------------
Files:
- aideen-backbone/src/shaders/lm_train.wgsl
- aideen-backbone/src/gpu_lm_head.rs
- aideen-training-lab/src/trainer.rs

Role:
- Converts pooled hidden state (`hpooled` plus optional Assoc contribution) into sampled logits/probabilities, loss, and `dl_dh` for the DEQ/FPM/Assoc backward path.

Current circuit invariant:
- For each token row, sampled probabilities must satisfy `0 <= probs[t,k] <= 1` and `sum_k probs[t,k] ~= 1` before `lm_backprop_h_t_main` computes `dl_dh`.
- `probs` storage is the single source of truth for logits/probabilities in the current correctness corridor.
- The serial tid0 max/sum normalization is intentionally performance-sensitive but equation-correct; any future parallel version must reproduce the same row-sum/probability invariant before promotion.

Failure mode found:
- The previous workgroup reduction/cache path produced invalid probabilities (`row sum ~1e13`, many entries >1), while LM weights and pooled inputs were normal. This created huge `dl_dh`, which then poisoned Picard adjoints and Assoc/FPM backward.

Validated observation:
- Serial normalization restored `row0_sum ~= 1`, removed probabilities >1, and reduced `dl_dh`, `assoc_rhs`, and `adj_v_out` to small finite scales. Four-chunk validation kept active Assoc `val_rms` around `1e-3` without NaNs.

Open trace tasks:
- Re-parallelize only after adding an invariant check/probe that proves probability rows remain normalized for `num_samples=2048`.
- Trace why Assoc budget telemetry for s6/s7 can disagree with final `[ASSOC-STATE]` occupancy.

Assoc Read Confidence Path Update
---------------------------------
Files:
- aideen-block/src/shaders/deq_slot_attn_unified_clean.wgsl
- aideen-training-lab/src/trainer.rs
- aideen-backbone/src/fixed_point_memory_reasoning.rs
- aideen-backbone/src/gpu_lm_head.rs

Current invariant:
- Hard-read bank selection may choose one bank, but read confidence must come from query/key address match, not from the post-hard-read probability. Under `AIDEEN_ASSOC_CONF_READ=1`, the read multiplier is now based on pre-hard-read best address cosine clamped to `[0,1]`.
- `alpha_assoc` is stored in logit space because GPU forward/backward apply `sigmoid(alpha_assoc)`. Default initialization stores `logit(0.3)`, giving effective read mix 0.3.
- Direct Assoc-to-LM logit coupling is disabled by default (`AIDEEN_ASSOC_LOGIT_LAMBDA=0`) until promotion/read utility is validated. The env override remains for experiments.

Failure modes found:
- Previous `alpha_assoc=0.3` produced effective read mix `sigmoid(0.3)=0.574`, stronger than the documented 0.3.
- Previous `ENABLE_ASSOC_CONF_READ` was inert under hard-read because `read_max_prob=1` for the selected bank even when the address match was weak.
- Positive Assoc LM-logit coupling degraded the short real-text corridor while memory promotion remains immature.

Open trace tasks:
- Add durable promotion/salience policy so hard/confident reads select predictive bindings, not merely first-admitted literal transitions.
- Retest positive Assoc LM-logit coupling only after selected bindings become useful under confidence-gated reads.

Assoc write admission - experimental salience replacement
---------------------------------------------------------
Path: `deq_slot_attn_unified_clean.wgsl` write phase, after slot/bank selection and before `AssocHist` metadata is emitted.
Source values: `best_cos_safe` measures key similarity to existing banks; `best_prev_value_cos` measures whether the candidate value duplicates previous hidden source; `occupied_count` and `min_usage/min_usage_bank` summarize current bank occupancy metadata.
Invariant: when the experimental salience path is disabled, `usage` remains committed write mass and no replacement occurs. When enabled, metadata is interpreted as salience and replacement uses `overwrite_bank=1`; forward key/value update clears prior bank contribution, matching backward's overwrite handling.
Failure mode addressed: bounded banks preserved first-arrival bindings even when later bindings had stronger evidence, so long-context memory could become full but not useful.

Assoc read backward parity
--------------------------
Path: `aideen-backbone/src/shaders/fused_fpm_retain_bwd.wgsl`, read-backward section after recovering per-bank address scores.
Forward invariant: when `ENABLE_ASSOC_HARD_READ=true`, bank weights are one-hot; when `ENABLE_ASSOC_CONF_READ=1`, value injection is scaled by `slot_read_allowed * clamp(best_address_score / ASSOC_READ_BETA, 0, 1)`.
Backward invariant: the fused backward path now receives `ENABLE_ASSOC_HARD_READ` and mirrors the hard one-hot selection. Confidence uses the same address-score scalar. This avoids training an inactive softmax read allocator when forward is hard-read.

Assoc write backward parity
---------------------------
Path: `fused_fpm_retain_bwd.wgsl`, durable associative memory backward.
Forward invariant: `effective_write_mass = effective_bind_gate` when `ENABLE_ASSOC_LINEAR_WRITE=true`, otherwise `effective_bind_gate^2`; writes below `ASSOC_WRITE_MIN_MASS` become zero; replacement overwrite sets prior keep gate to zero.
Backward invariant: fused backward now uses the same write mass, the matching derivative `d mass / d effective_bind_gate`, and zero keep gradient when `overwrite_bank=1`.

LM sampled-negative determinism invariant
-----------------------------------------
Files:

Role:
- `GpuLmHeadTrainer::build_sampled_indices` constructs the sampled-softmax vocabulary slice consumed by `lm_probs_main`, `lm_backprop_h_t_main`, and `lm_update_main`.

Invariant:
- For stabilization runs, sampled indices must be reproducible for a fixed LM sample seed, `step_t`, targets, vocabulary size, and sample capacity.

Validated status:
- `GpuLmHeadTrainer` now uses `AIDEEN_LM_SAMPLE_SEED`, falling back to `AIDEEN_TRAIN_SEED`, to generate deterministic sampled-softmax negatives from a counter/hash function keyed by `step_t`.
- The sampled set is no longer drawn from `thread_rng`; this removes an uncontrolled stochastic input from the LM loss and `dl_dh` path.
- Remaining caveat: 16-chunk repeated runs are stable but not bit-identical, so another non-bitwise path remains outside sampled-negative construction.

Assoc bank-capacity trace correction
------------------------------------
- Forward Assoc bank count is controlled by `AIDEEN_ASSOC_BANKS` in `RustDeqBridge` and `GpuDeqBackend`, then passed as WGSL override `ASSOC_BANKS` into `deq_slot_attn_unified_clean.wgsl`.
- Forward bank placement/read local arrays must be at least `ASSOC_BANKS`; current experimental supported cap is 32 banks.
- Backward Assoc in `fused_fpm_retain_bwd.wgsl` historically copied one full slot state into `assoc_b_state[4096]` and accumulated `assoc_gb[4096]`. Required floats are `ASSOC_BANKS * (ASSOC_RANK + d_model + 1)`. For `d_model=512`, even 16 banks need 8720 floats, so large-bank backward cannot safely use this workgroup cache.
- Current invariant: scalable bank-count experiments may validate forward memory/layout up to 32 banks, but Assoc backward is skipped when the slot state exceeds 4096 floats. Full scalable training requires storage-backed or tiled Assoc adjoints.

Large-bank Assoc key-write backward
-----------------------------------
Path: `aideen-backbone/src/shaders/fused_fpm_retain_bwd.wgsl`, branch `assoc_slot_stride > ASSOC_BWD_STATE_CAP`.
Forward source of truth: `deq_slot_attn_unified_clean.wgsl` stores per-token pre-write banks in `AssocHist` plus metadata `[chosen_bank, allow_write, transition_gate, overwrite_bank]` after the full-sequence coordinate fix.
Backward invariant now covered: large-bank read score gradients write a compact key-only adjoint `assoc_gb[bank * ASSOC_RANK + r]`. For the bank selected by forward metadata, the adjoint is propagated through the active key-write equation into `W_k_assoc` using the same write mass and overwrite keep semantics as forward. This avoids copying `[key|value|usage]` for every bank into the 4096-float workgroup cache.
Still out of scope: full value adjoint, event/librarian/replacement-policy gradients, and exact `PrevHStarBuf` key-source parity. The current large-bank key backward mirrors the existing backward source convention (`S_in[t-1]`) to avoid adding a new buffer dependency mid-corridor.

FPM vs Assoc slot telemetry split
---------------------------------
Path: `aideen-block/src/shaders/deq_slot_attn_unified_clean.wgsl`.
Invariant: slots `slot_idx < h_slots/2` run the base FPM plastic write block; slots `slot_idx >= h_slots/2` are Assoc slots and skip that base block via `!is_assoc_slot`. Assoc slots can still consolidate a slow contextual trace into FPM state inside the Assoc write block (`ASSOC_TO_FPM_SCALE * effective_write_mass`), but current `[FPM-PATH] write/retain` telemetry is sourced from the base FPM write diagnostics, so zeros on slots 4..7 do not by themselves prove that Assoc-to-FPM consolidation is inactive.
Failure-risk: real-text logs can look like half the slots are dead if interpreted only through FPM write counters. Need explicit Assoc-to-FPM bridge telemetry before deciding whether contextual consolidation from Assoc slots is weak or merely unobserved.

Assoc-to-FPM bridge observability
---------------------------------
Path: `deq_slot_attn_unified_clean.wgsl` Assoc write block, after exact key/value bank write.
Observed equation: `consolidate = ASSOC_TO_FPM_SCALE * clamp(assoc_write_gate_token, 0, ASSOC_WRITE_CAP) * effective_write_mass`; `fpm_m_cache` then moves toward normalized `H_curr` trace by that consolidate amount.
Telemetry: `[ASSOC-BRIDGE]` now reports total-average and active-average bridge mass/consolidation/delta per Assoc slot. This is necessary because base `[FPM-PATH] write(...)` counters intentionally exclude Assoc slots and therefore cannot diagnose whether the Assoc contextual bridge is active.
Current real-text observation: bridge is wired and finite, but sparse/low-energy on average. Active writes can reach the configured max consolidate, while total average is dominated by zero/near-zero write mass.

Assoc-to-FPM scale control
--------------------------
`ASSOC_TO_FPM_SCALE` is now a shader override supplied by `AIDEEN_ASSOC_TO_FPM_SCALE` with default `0.02`. It directly multiplies the Assoc contextual consolidation equation and is therefore an explicit experimental control surface, not a hidden default behavior change.

Direct Assoc-to-LM path check
-----------------------------
- Real train path: `Trainer` calls `GpuLmHeadTrainer::train_step_no_readback(...)` with `h_src = gpu.bridge.hpooled_buf` and `assoc_src = gpu.bridge.assoc_pooled_buf`.
- `gpu_lm_head.rs` copies `assoc_src` into LM `dl_dh_temp_buf` before the forward LM probability pass.
- In `lm_train.wgsl`, direct Assoc logits are disabled by default because `AIDEEN_ASSOC_LOGIT_LAMBDA` defaults to `0.0`.
- If enabled, the sampled logit equation becomes `logit(v,t) = b[v] + W_lm[v] dot RMSNorm(h_t) + lambda * Emb[v] dot AssocRead_t`.
- Backward then writes the Assoc RHS into `s_h_rms`/`assoc_grad_buf`: `dL/dAssocRead_t = sum_k (p_k - y_k) * lambda * Emb[k]`.
- Trainer passes both `dl_dh_buf` and `assoc_grad_buf()` to `run_staged_adjoint_picard_no_readback`, so the direct Assoc LM path is gradient-coupled when lambda is nonzero.
- Current serious-training baseline keeps this lambda at `0.0` because the first short real-text validation did not beat the no-bridge baseline.

Current promoted end-to-end training circuit
--------------------------------------------
Promoted serious-prep path:
`token ids -> GPU embeddings -> DEQ slot solve -> FPM slots + Assoc slots -> hpooled/assoc_pooled -> GPU sampled LM head -> loss -> LM backward -> staged DEQ adjoint -> GPU weight/state updates`.

What is active:
- Embedding training is active unless `--freeze-emb` is explicitly passed; promoted runs do not freeze it.
- DEQ/FPM/Assoc forward is active on GPU/Metal.
- LM head forward/backward is active through `GpuLmHeadTrainer::train_step_no_readback`.
- DEQ adjoint receives `dl_dh_buf` and, when direct Assoc LM lambda is nonzero, `assoc_grad_buf`; in the promoted baseline lambda is zero, so the direct Assoc LM RHS is intentionally inactive.
- Assoc slots use explicit memory banks in the forward path with 32-bank support in the promoted corridor.

What is intentionally inactive in the promoted baseline:
- Direct Assoc-to-LM logit contribution: `AIDEEN_ASSOC_LOGIT_LAMBDA=0.0`.
- Assoc-to-FPM bridge contribution: `AIDEEN_ASSOC_TO_FPM_SCALE=0.0`.
- Held-out validation updates: skipped by design because `eval_mode=true` sets LM lr to zero and returns before optimizer/adjoint updates.

Known partial areas:
- Large-bank Assoc backward has scalable key-write coverage, but full value adjoint and replacement/librarian/event gradients remain pending.
- Base FPM write telemetry intentionally reports zeros on Assoc slots because those slots do not use the base FPM write block; Assoc-specific bridge telemetry is the correct observer for contextual consolidation.

Large-bank Assoc selected value adjoint
---------------------------------------
Path: `aideen-backbone/src/shaders/fused_fpm_retain_bwd.wgsl`, `assoc_slot_stride > ASSOC_BWD_STATE_CAP` branch.
Forward equation: selected bank value is updated as `bank_value <- value_keep_gate * bank_value + value_write_mass * normalized_current_value`.
Backward addition: because full `ASSOC_BANKS * d_model` value adjoints do not fit in workgroup storage at 32 banks, the large-bank branch computes only the forward-selected bank's value adjoint from the read equation: `dL/dbank_value_selected = alpha_assoc * assoc_read_conf * p_selected * g_ctx`. This selected adjoint contributes to the write-mass/event-gate gradient through `(normalized_current_value - previous_bank_value)`.
Invariant: no full-bank value cache is introduced; storage stays compact and proportional to `d_model + ASSOC_RANK`, not `ASSOC_BANKS * d_model`.
Diagnostic: `[ASSOC-BWD-STEP] value_adj_sum` reports nonzero selected value adjoint mass on Assoc slots in debug runs.
