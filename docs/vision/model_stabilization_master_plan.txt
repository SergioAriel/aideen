Change gate - AssocHist segmented global indexing
-------------------------------------------------
1. Target failure mode: in segmented forward, AssocHist snapshots are written using local segment index `t` and `shape.token_count`, while backward reads them using full-sequence `t` and `params.seq_len`. Long sequences/chunks can therefore train against stale or wrong pre-write banks.
2. Structural rationale: AssocHist is the forward/backward bridge for associative bank state. Its invariant is: forward snapshot index == backward token index == `(batch * full_seq_len + global_t) * h_slots + slot`. Forward already computes `global_t = shape.token_start + t` for S_in/H/Scratch; AssocHist must use the same global coordinate.
3. Anti-patch check: this does not tune gates, losses, or scales. It restores the required layout/ownership invariant at the buffer boundary where the mismatch originates, and remains valid across segment lengths, seeds, and profiles.
4. Validation plan: cargo check; then rerun the baseline corridor log. Success requires no NaNs and changed/cleaner backward behavior without relying on oracle flags. If metric remains random, proceed to oracle write/read/LM diagnostics with the history invariant fixed.
Classification target: layout/invariant, default-stable candidate if validation passes.

Investigation gate - learned temporal write selectivity after oracle boundary
----------------------------------------------------------------------------
1. Target failure mode: with full-document capacity and default Assoc->LM logit bridge, oracle write positions solve long gaps, but normal writes contaminate banks by writing fillers/query transitions. Linear write mass improves oracle-mask cases but no-oracle long gaps remain random.
2. Structural rationale: explicit Assoc memory needs a persistence policy that decides which transitions become durable bindings. The required invariant is not merely nonzero write mass; it is high write mass on binding transitions and low write mass on irrelevant filler/query transitions, while preserving readable bank values.
3. Anti-patch check: do not tune loss or clamp output to hide contamination. The next change, if any, must affect the write-selection mechanism at its origin or fix an existing selector/gradient path that is currently ineffective.

Change gate - protect occupied Assoc banks from filler overwrites
-----------------------------------------------------------------
1. Target failure mode: no-oracle Assoc writes still contaminate long-gap memory because every token transition writes into the finite bank set. With one/few banks, the original key->value binding is overwritten or averaged away by fillers before query.
2. Structural rationale: explicit Assoc banks are durable literal bindings, not a dense recurrent accumulator. A write into an already occupied bank should require either an empty-bank allocation, a reuse/match decision, or another strong replacement policy. In the absence of that, write mass must be reduced by available capacity `(1-usage)` so occupied banks preserve stored evidence.
3. Anti-patch check: this fixes the ownership/lifetime invariant of finite banks at the write equation. It does not clamp logits or use benchmark token IDs. It remains valid for generic finite associative memory: preserve occupied bindings unless replacement is justified.
4. Validation plan: cargo check; run no-oracle full-document probe with linear write and occupied protection. Success requires lower long-gap loss and telemetry showing reduced effective writes after bank occupation. If too conservative, test with multiple banks or replacement threshold rather than disabling the invariant.
Classification target: architecture/dynamics default candidate only after no-oracle validation.

Change gate - promote linear Assoc write mass
---------------------------------------------
1. Target failure mode: with occupied-bank protection active, quadratic write mass (`g*g`) underwrites the initial binding, leaving long-gap recall only partial (~2.9 loss). Linear write mass with the same protection solves no-oracle long gaps (~0.95 loss).
2. Structural rationale: Assoc is a literal binding memory. Once the selector decides to commit a binding into an empty bank, the write mass should correspond to selector confidence, not confidence squared. Squaring is appropriate as a conservative damping patch, but here it starves the only durable write before the bank becomes protected.
3. Anti-patch check: this changes the write equation at the origin of underwritten bindings. It does not use token IDs, oracle positions, or output clamps. It remains bounded by bank protection and existing gate factors.
4. Validation plan: set ENABLE_ASSOC_LINEAR_WRITE default true; cargo check; rerun no-oracle full-document corridor with no `AIDEEN_ASSOC_LINEAR_WRITE` env. Success requires gap0/64/128/256 materially below random and no NaNs.
Classification target: dynamics/default-stable candidate after validation.

Change gate - suppress writes from already-stored value sources
---------------------------------------------------------------
1. Target failure mode: with multiple key/value pairs, occupied-bank protection preserves banks but still stores alternating bad transitions such as value->next_key before banks are full. This consumes finite literal capacity and leaves only a subset of queryable key->value bindings.
2. Structural rationale: in explicit associative memory, a token/vector already stored as a bank value should not automatically become the source key for another durable binding on the next step. Suppressing writes when the previous source matches an existing bank value prevents value-chain contamination while still allowing new key-like sources to allocate empty banks.
3. Anti-patch check: this uses only memory geometry (`PrevHStar` vs stored bank values), not benchmark token IDs, positions, or target labels. It restores a role invariant: values are retrievable payloads, not default write addresses, unless a stronger replacement/reuse policy later justifies it.
4. Validation plan: cargo check; run pairs_per_seq=4 with AIDEEN_ASSOC_BANKS=4 and full-document capacity. Success requires gap128/256 materially below random and audit showing preserved multiple bindings. Also rerun pairs=1 default corridor to ensure no regression.
Classification target: architecture/dynamics default candidate after validation.

Research alignment - external/scalable memory directions
-------------------------------------------------------
Sources reviewed:
- DNC / Neural Turing-style memory: content/location addressing, usage/allocation vectors, explicit external memory.
- Product-Key Memory: very large key-value memory accessed by structured/exact nearest-neighbor style lookup.
- kNN-LM / Memorizing Transformer / RETRO / LongMem: external or cached key-value/chunk retrieval for long-tail facts and long context.
- Transformer-XL / Compressive Transformer: segment recurrence and compressed long-range memory.
Implication for AIDEEN:
- Current Assoc hard-read behavior is aligned with kNN/key-value literal retrieval, not with soft attention over values.

Capacity roadmap - from replicated Assoc to scalable lanes
---------------------------------------------------------
Problem:
- Current validated corridor uses h_slots=4 and ASSOC_BANKS<=8 with Assoc slots replicated. Exact capacity is therefore banks, not banks × assoc_slots.
- Larger goals such as 32 banks and 16 slots require distinguishing technical limits from architectural policy.
Current technical limits:
- WGSL local bank arrays are fixed length 8, so ASSOC_BANKS>8 is currently unsupported without shader refactor.
- Assoc slots are the upper half of h_slots. With h_slots=4, there are 2 Assoc slots, but default slot-owner is off, so they replicate the same memory.
Target architecture:
- Local exact capacity should become assoc_lanes × banks_per_lane once slot-owner/lane routing is validated.
- FPM should contextualize/compress over time, not carry every exact binding.
- Persistent Assoc should promote selected exact bindings across segments.
Next focus:
1. Validate slot-owner/lane routing as a capacity multiplier with current 8-bank shader limit.
2. If lane routing works, test h_slots=8 or 16 with banks<=8 to estimate practical capacity/cost.
3. Only then refactor shader local arrays to support banks=16/32, with layout checks and performance measurements.

----------------------------------------------------------
1. Target failure mode: after large-bank read backward, future reads can train `W_q_assoc` but do not push gradient back into the key that was written by `W_k_assoc`. Thus large-bank training still lacks the write-key learning path.
